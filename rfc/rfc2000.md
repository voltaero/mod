# RFC 2000: `rustforge` — Engine Mod Dev CLI
**Status:** Draft  
**Author:** @ign-styly  
**Last updated:** 2026-03-12  

## Summary
`rustforge` is a CLI tool to make developing Engine mods straightforward and reproducible by:

1. Deriving the required Engine version from the git commit pinned by `enginelib` in the mod project’s `Cargo.lock`
2. Cloning/fetching the Engine repo and creating a local checkout for that exact commit as an internal implementation detail of `rustforge dev`
3. Building/packaging the mod into a `.rf` artifact (a tar archive renamed to `.rf`, containing `mod.so`/`mod.dll`) for:
   - **Testing:** install into the Engine checkout and run the Engine
   - **Final build:** produce a distributable `.rf` file under `dist/`
4. Optionally watching the mod project for changes and restarting the Engine automatically during local development

This RFC standardizes:
- The tool/binary name: **`rustforge`**
- Engine checkouts are cached **project-locally** under **`.rustforge/`**
- Mod artifacts are **only `*.rf`**
- The public CLI surface is centered on `status`, `mod`, and `dev`

## Motivation / Problem statement
Engine mods are dynamic libraries that must be compatible with the Engine binary they’re loaded into. Compatibility typically depends on:
- The Engine/`enginelib` API version used by the Engine binary
- The Rust compiler version used to build both

Without tooling, a developer must manually:
- Determine which Engine commit corresponds to the `enginelib` version their mod is built against
- Clone/fetch/reset the Engine repo to that commit
- Build the Engine binaries
- Build the mod `cdylib`
- Rename/copy it into the correct package format and place it where the Engine will load it

`rustforge` automates the above with a single source of truth: the mod project’s `Cargo.lock`.

## Goals
- Determine Engine commit from the `Cargo.lock` entry for `enginelib` (`git+...#<sha>`).
- Cache Engine repo locally under `.rustforge/engine/` and only fetch when needed.
- Provide one “happy path” command for dev: **build + package + install + run**.
- Provide an optional watcher mode for dev that rebuilds and restarts the Engine on mod changes.
- Package output is `*.rf`, where `.rf` is a tar archive whose root contains:
  - `mod.so` on unix (including macOS)
  - `mod.dll` on Windows
- Support packaging for the common desktop targets Linux, macOS, and Windows from a single command.

## Non-goals
- Not a general-purpose Rust build system.
- Not a mod publishing platform.
- Not a replacement for Cargo; it orchestrates Cargo/Git.

## Terminology
- **Mod project:** the repo where you run `rustforge` (this repo is an example mod project).
- **Engine commit:** the git SHA pinned by `enginelib` in the mod project’s `Cargo.lock`.
- **`.rf` artifact:** a tar archive renamed to `.rf`, loadable by Engine when placed in `./mods`.

## Design

### Source of truth: deriving the Engine version from `Cargo.lock`
The Engine commit is derived from the mod project’s lockfile, not from a human-maintained config.

#### Algorithm
1. Locate lockfile:
   - Default path: `./Cargo.lock` adjacent to `./Cargo.toml`
   - If missing: error with remediation (e.g. run `cargo generate-lockfile` or `cargo build`)

2. Parse `Cargo.lock` (Cargo lockfile TOML; v3 and v4 are accepted):
   - Find a `[[package]]` entry with `name = "enginelib"`
   - Read its `source` field
   - Require a git source of the form:
     - `source = "git+<repo_url>#<full_sha>"`
   - Allow additional query parameters before the `#`
   - Always take the substring after the **final** `#` as the commit SHA

3. Outputs:
   - `engine.repo_url` = `<repo_url>` (strip the `git+` prefix)
   - `engine.commit` = `<full_sha>`
   - `engine.commit_short` = first 12 chars (display only)

#### Failure modes
- No `enginelib` entry found:
  - Error: “`enginelib` not found in Cargo.lock; is this a mod project lockfile?”
- `enginelib` is not git-sourced (no `git+...#...`):
  - Error: “Only git-pinned Engine/enginelib is supported; update dependency to a git source.”
- Multiple `enginelib` matches:
  - Error: “multiple `enginelib` entries found in Cargo.lock; add configuration to disambiguate before continuing”

#### Example
In a lockfile you may see a git source like:

```toml
name = "enginelib"
source = "git+https://github.com/voltaero/engine.git#f9ddcc8e9eff65906f9000ec16638e933ebac20b"
```

In that case:
- `engine.repo_url = "https://github.com/voltaero/engine.git"`
- `engine.commit = "f9ddcc8e9eff65906f9000ec16638e933ebac20b"`

### Engine repo cache + checkout management
The Engine repo is cached within the mod project under `.rustforge/`, using a bare mirror plus detached worktrees for specific commits.

#### On-disk layout
- `.rustforge/engine/mirror.git/` — bare mirror clone of the Engine repo
- `.rustforge/engine/worktrees/<engine_commit>/` — detached worktree checkout of the exact commit
- (Optional) `.rustforge/engine/worktrees/<engine_commit>/.rustforge-build/` — tool-owned build outputs
  - If not used, regular Cargo `target/` under the worktree is acceptable.

#### Sync rules
- The mirror’s remote URL defaults to `engine.repo_url` derived from `Cargo.lock` (unless overridden).
- `rustforge` must ensure the Engine checkout before any `dev` run:
  1. Create mirror if missing:
     - `git clone --mirror <repo_url> .rustforge/engine/mirror.git`
  2. If mirror exists, fetch updates as needed:
     - `git -C .rustforge/engine/mirror.git fetch --prune`
  3. Ensure the required commit exists locally; if not, fetch it explicitly.
  4. Create a detached worktree if missing:
     - `git -C .rustforge/engine/mirror.git worktree add --detach .rustforge/engine/worktrees/<sha> <sha>`

#### Read-only worktrees
Worktrees are treated as read-only. If a worktree is dirty or not at the expected commit, `rustforge` may delete and recreate it.

### Mod build + `.rf` packaging format

#### Build
- Dev/test build:
  - `cargo build`
- Final build:
  - `cargo build --release`
- `--target <triple>` must be supported and passed through to Cargo for cross builds.

#### Locating the built `cdylib`
`rustforge` uses `cargo metadata` to discover:
- Cargo `target_directory`
- The mod package name (and thus expected artifact naming)

Expected artifact filenames:
- Linux: `lib<crate>.so`
- macOS: `lib<crate>.dylib` (but will be packaged as `mod.so`)
- Windows: `<crate>.dll`

#### Staging rename (required)
The `.rf` archive root must contain:
- unix (incl. macOS): `mod.so`
- windows: `mod.dll`

To accomplish this, `rustforge` copies the built artifact into a staging directory and renames it accordingly.

#### `.rf` archive creation
`.rf` is a tar archive renamed to `.rf` (only the extension differs).

Archive requirements:
- `mod.so`/`mod.dll` is at the **tar root** (no subdirectory)
- Additional files are allowed, but must not move/rename `mod.*`

Output naming:
- Test artifact:
  - `.rustforge/out/<mod_id>.rf`
  - `<mod_id>` comes from config; defaults to Cargo package name
- Final artifact:
  - `dist/<mod_id>-<mod_version>-engine-<shortsha>.rf`
  - If `mod_version` is unknown/unconfigured, omit that segment.
- Multi-platform package output (`--all-platforms`):
  - Debug/test: `.rustforge/out/<mod_id>-<target>.rf`
  - Final: `dist/<mod_id>-<mod_version>-<target>-engine-<shortsha>.rf`

Install-for-test:
- Copy/symlink the produced `.rf` to:
  - `.rustforge/engine/worktrees/<sha>/mods/<mod_id>.rf`

## CLI surface area

### `rustforge status`
Prints:
- Derived Engine repo URL + commit (from `Cargo.lock`)
- Whether `.rustforge/engine/mirror.git` exists
- Whether `.rustforge/engine/worktrees/<sha>` exists
- Computed output paths for test/final `.rf` artifacts

### `rustforge mod build [--release] [--target <triple>]`
Builds the mod project (current repo).

### `rustforge mod package [--release] [--target <triple>] [--all-platforms]`
Builds (if needed), then:
- Stages `mod.so`/`mod.dll`
- Creates `.rf` as specified
- `--target <triple>` packages a single explicit target
- `--all-platforms` packages all default desktop targets:
  - `x86_64-unknown-linux-gnu`
  - `x86_64-apple-darwin`
  - `x86_64-pc-windows-msvc`
- `--all-platforms` and `--target` are mutually exclusive

### `rustforge dev [--release-engine] [--release-mod] [--target <triple>] [--hot]`
The “one command” flow:
1. Derive Engine commit from `Cargo.lock`
2. Ensure the Engine mirror/worktree exist for that commit
3. Build Engine (debug by default; release if `--release-engine`)
4. Build mod (debug by default; release if `--release-mod`)
5. Package `.rf`
6. Install into Engine worktree `./mods`
7. Run Engine with cwd = Engine worktree root

#### Reload behavior
`rustforge dev --hot` watches the mod project for changes, ignoring generated directories such as `.rustforge/`, `dist/`, and `target/`. On a detected change it:
1. Rebuilds the mod
2. Repackages the `.rf`
3. Reinstalls it into the Engine worktree
4. Restarts the Engine process

If rebuild or packaging fails, the existing Engine process remains running and the error is reported.

## Configuration

### Discovery order
1. `[package.metadata.rustforge]` in `Cargo.toml` (preferred)
2. `[metadata.rustforge]` in `Cargo.toml` (legacy compatibility)
3. Optional `rustforge.toml` at repo root (only if needed for non-Cargo setups)

### Keys and defaults
- `mod_id` (string)
  - Default: Cargo package name
- `engine.bin` (string)
  - Default: `server`
- `engine.features` (array of strings)
  - Default: `[]`
- `engine.repo_url` (string)
  - Default: derived from `Cargo.lock`
- `paths.engine_root` (string)
  - Default: `.rustforge/engine/worktrees/<sha>`
- `paths.out_dir` (string)
  - Default: `.rustforge/out` (test artifacts)
  - Final artifacts default to `dist/`

### Invariant: config must not change the Engine commit
The Engine commit is always derived from `Cargo.lock`. Configuration can override the repo URL (rare) but must not override the commit.

## Testing / Acceptance criteria (for the future implementation)

### Lockfile parse
- Given a lockfile with `enginelib` git source, `rustforge status` prints repo URL + SHA.
- Missing lockfile produces a clear error with remediation.

### Engine checkout management
- If `.rustforge/engine/mirror.git` is absent, it is created.
- If the required commit is not present locally, it is fetched.
- Worktree `.rustforge/engine/worktrees/<sha>` exists afterward at the exact commit.

### Packaging
- Produces `<mod_id>.rf` containing `mod.so`/`mod.dll` at tar root.
- `rustforge mod package --all-platforms` produces one artifact per default target with the target triple in the filename.
- Installing places the `.rf` in Engine worktree `./mods/`.

### Dev flow
- `rustforge dev` completes end-to-end on a clean checkout (assuming Rust + git + tar available).
- `rustforge dev --hot` rebuilds, reinstalls, and restarts the Engine after a mod source change.

## Assumptions
- `.rf` is a tar archive renamed; only the extension differs.
- Engine loads modules from `./mods` under the Engine working directory.
- Engine and mod must match the API version implied by the `enginelib` commit.
- Git and Cargo are available on developer machines.

## Implementation sketch (not part of this RFC’s scope)
Recommended implementation choices:
- Implement `rustforge` using the existing crate in `rustforge/` (replace the current hello-world).
- Use `cargo metadata` to reliably discover target directories and artifact naming.
- Use a bare git mirror plus worktrees for fast, reproducible checkouts by commit.
